{# ── Application Kit ─────────────────────────────────────── #}

Fit assessment

If a posting's stated salary range tops out below this amount, the fit assessment will flag it as a concern.

The get_kit_instructions workflow

When Claude calls get_kit_instructions via the MCP connector, it receives a complete six-step playbook that drives every application kit build. Here is what each step does:

Step 0 Fit Assessment

Before any documents are written, Claude compares the live posting against the candidate profile and outputs a one-line verdict (Strong Fit, Partial Fit, or Stretch). It flags concerns — salary below the floor set above, hard requirements not met, overqualification, or location conflicts — and closes with a recommendation to Proceed, Proceed with caveats, or Consider skipping. If the recommendation is "Consider skipping," Claude stops and asks before continuing.

Step 1 Gather Reference Documents

Claude pulls the candidate's source documents. When the MCP connector is active it calls get_candidate_assets() to retrieve uploaded resumes and templates. In Cowork mode it reads the .docx files directly from the Resumes folder. If neither is available it falls back to the embedded candidate profile.

Step 2 Fetch the Live Posting & Research the Company

Claude fetches the current posting URL to get the full, up-to-date job description (not just the captured snapshot). It also runs a web search for recent company news, market position, and team context. Findings are summarized in a Research Notes block that appears at the top of every kit output.

Step 3 Build the Application Package

Claude extracts the 10-15 most important ATS keywords from the posting, marks each as Present, Partial, Absent, or a Gap against the profile, and notes how each will be incorporated. It then produces five deliverables using the live posting and research findings: a tailored ATS resume, a cover letter (under 300 words referencing a specific company detail), an application email, three follow-up and thank-you emails, and three role-specific interview questions. All writing follows a strict style guide: no em-dashes, no AI clichés, no fabricated facts.

Step 4 Save Artifacts

The completed kit is saved in two formats to Application Kits/ under the project folder: a Markdown file (kit-output-{slug}.md) for easy reference, and a formatted Word document (kit-output-{slug}.docx) with proper headings. Both are presented to the user as downloadable files.

Step 5 Push to Job Squire

If the MCP connector is active and the kit header contains a Job ID, Claude calls save_kit() to attach the full kit markdown (including the fit assessment) back to the job record in Job Squire. The kit then appears on the job detail page. If no Job ID is present, this step is skipped and noted.

To trigger the workflow from Claude, open a chat with the Job Squire connector active and say: "Build an application kit for job ID [number]." Claude will call get_kit_instructions automatically and follow the steps above. You can also build a kit from the Application Kits tab by clicking Build Kit on any saved job.
